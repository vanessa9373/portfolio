How I Built a Multi-Account AWS Landing Zone from Scratch

Jun 27, 2026 10 min read AWS Architecture Governance Terraform

I built a Multi-Account AWS Landing Zone from scratch to understand how enterprises securely manage AWS at scale.

Most AWS users begin with a single account. Large organizations operate dozens or even hundreds of accounts. A Landing Zone provides the governance, security, networking, and identity foundation needed to manage that environment consistently. Here's the architecture I built and what I learned.

AWS Organizations

I organized the environment into four Organizational Units (OUs):

  • 🔒 Security — centralized GuardDuty, Security Hub, and a dedicated Log Archive account for centralized monitoring and threat detection.
  • 🌐 Infrastructure — AWS Transit Gateway, shared DNS services, and centralized egress inspection.
  • 🛠 Development — sandbox accounts with relaxed but still governed policies for testing and experimentation.
  • 🚀 Production — customer-facing workloads with strict security controls and the highest level of governance.

Every AWS account belongs to one of these OUs. The OU determines what that account can ever do — enforced by Service Control Policies, not IAM. This is the critical distinction.

Service Control Policies (SCPs)

While IAM defines what users and roles are allowed to do, SCPs define the maximum permissions any account can receive — even if AdministratorAccess is assigned. There is no way around an SCP from inside the account.

I implemented eight SCPs across the organization:

  • DenyRootUser — restricts root account usage except for emergency break-glass situations. Root credentials should never be used after initial setup.
  • RequireIMDSv2 — enforces Instance Metadata Service v2 on every EC2 instance, blocking the SSRF attack vector that compromised Capital One in 2019.
  • AllowedRegionsOnly — restricts deployments to approved AWS Regions, preventing data residency violations and rogue resource creation.
  • DenyPublicS3 — prevents any S3 bucket from becoming publicly accessible, organization-wide and uncircumventable.
  • RequireEncryption — requires encryption for Amazon EBS volumes and Amazon RDS databases at creation. Unencrypted CreateVolume calls are denied.
  • DenyCloudTrailDisable — prevents CloudTrail from being disabled or modified. The audit trail cannot be tampered with by anyone inside any account.
  • RequireMFAForConsole — requires MFA for every console login across the organization.
  • DenyLargeInstanceTypes — prevents development environments from launching expensive GPU or oversized EC2 instances.

The key insight: SCPs are preventative, not detective. They stop the mistake before it happens — not three hours later when GuardDuty fires an alert.

AWS Transit Gateway

Without Transit Gateway, connecting VPCs requires a peering mesh that grows as O(n²). With 10 accounts you need 45 peering connections. With 20 accounts, 190. With 50, over 1,200. Each connection is a separate routing configuration to manage.

Transit Gateway replaces all of that with hub-and-spoke — every VPC connects to one central gateway. Routing is managed in one place. Adding a new account means adding one attachment, not reconfiguring every existing peer.

Route table segmentation enforces hard boundaries: Production VPCs cannot route to Development VPCs at the network layer, not just at the IAM layer. All internet egress flows through an Inspection VPC with network firewall rules.

One non-negotiable constraint: plan your CIDR ranges before creating a single subnet. VPC CIDRs cannot be changed after creation, and overlapping ranges block Transit Gateway routing entirely.

IAM Identity Center

With 10+ accounts, managing IAM users in each account separately is operationally unsustainable — a separate set of credentials, passwords, and access keys in every account is both a compliance risk and a security liability.

IAM Identity Center provides a single identity source across every account. I configured SAML 2.0 federation with MFA enforced at the IdP level. Permission Sets define what each role can do — a developer gets read-only access to Production and full access to their Development account, managed centrally and applied everywhere simultaneously via Terraform.

When a team member leaves, one action in IAM Identity Center revokes access across all accounts instantly. No hunting for forgotten IAM users.

Centralized Security

All security tooling runs from the Security account and monitors every member account:

  • GuardDuty — delegated administration from the Security account. ML-based threat detection runs in every member account, all findings aggregate in one place. A compromised EC2 in any account appears on the Security team's dashboard within minutes.
  • Security Hub — aggregates findings from GuardDuty, Config, Inspector, and Macie across all accounts. The CIS AWS Benchmark score gives a single compliance number for the entire organization.
  • CloudTrail — org-wide trail ships all API activity to an S3 bucket in the Log Archive account with MFA-Delete and Object Lock enabled. The DenyCloudTrailDisable SCP ensures the trail can never be turned off — not even by an account administrator.

Key Lessons Learned

1. The Management Account is not governed by SCPs. SCPs don't apply to the Management Account — it's ungoverned by design, which means it requires the strongest operational controls: hardware MFA, break-glass credentials, and no workloads ever running in that account.

2. Network segmentation is just as important as IAM. Strong IAM policies alone aren't enough. Proper Transit Gateway route table segmentation is essential to prevent lateral movement between environments. You can have perfect SCPs and still have Production-to-Development reachability if TGW routing isn't properly configured.

3. GuardDuty delegated administration must be configured explicitly. Member account findings don't flow to the Security account automatically. Delegated administration must be enabled — most setups miss this step and wonder why security alerts aren't showing up centrally.

What This Project Demonstrates

A governance framework that scales from 10 to 1,000 accounts without changing the core architecture. New teams get a compliant, isolated, network-connected account automatically. Security findings are centralized. The audit trail is immutable. No developer can accidentally affect production. Every account starts with encryption enforced, root access blocked, and logging enabled.

This project strengthened my understanding of enterprise AWS governance, multi-account architecture, centralized identity management, security controls, and scalable cloud networking — all foundational skills for Solutions Architect and Solutions Engineer roles.
